深入理解 JVM 内存模型与垃圾回收机制

Created2025-01-28|Updated2026-05-07
|Word Count:2k|Reading Time:6mins|Post Views:

如果 Java 是一门功夫，那么 JVM 就是内功心法。不懂 JVM，写了再多代码也只是停留在「拳脚功夫」的层面。而 JVM 中最核心、最常被面试官问起、也最影响系统性能的部分，就是内存模型与垃圾回收机制。

这篇文章不打算抄写《Java 虚拟机规范》的条文，而是希望通过一个清晰的脉络，帮你建立起对 JVM 内存管理的整体认知。

JVM 运行时数据区域

JVM 在运行时将内存划分为若干个不同的数据区域，各有各的职责和生命周期。

程序计数器（Program Counter Register）

这是一块很小的内存空间，可以看作是当前线程所执行字节码的行号指示器。线程私有，生命周期与线程相同。如果线程执行的是 Java 方法，计数器记录的是正在执行的虚拟机字节码指令地址；如果是 Native 方法，则计数器值为空（Undefined）。

Java 虚拟机栈（Java Virtual Machine Stack）

每个方法执行时都会创建一个栈帧（Stack Frame），用于存储局部变量表、操作数栈、动态链接、方法出口等信息。每个方法从调用到执行完成的过程，就对应着一个栈帧在虚拟机栈中入栈到出栈的过程。

栈是面试高频考点，主要关注两个异常：

  • StackOverflowError：线程请求的栈深度大于虚拟机所允许的深度
  • OutOfMemoryError：虚拟机栈动态扩展时无法申请到足够内存

本地方法栈（Native Method Stack）

与虚拟机栈类似，区别在于虚拟机栈为 Java 方法服务，而本地方法栈为虚拟机使用到的 Native 方法服务。

Java 堆（Java Heap）

堆是 JVM 所管理的内存中最大的一块，所有线程共享。它的唯一目的就是存放对象实例——几乎所有的对象都在这里分配内存。

堆在物理上可以不连续，但在逻辑上连续。按照分代收集理论，堆可以分为：

方法区（Method Area）

用于存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据。在 Java 8 中，永久代（PermGen）被移除，取而代之的是元空间（Metaspace），使用本地内存而非 JVM 堆内存。

运行时常量池（Runtime Constant Pool）

属于方法区的一部分，用于存放编译期生成的各种字面量和符号引用。

垃圾回收（GC）的底层逻辑

许多同学一提 GC 就开始背算法，但实际上理解 GC 的核心在于理解三件事：

  1. 哪些内存可以回收？ → 存活分析算法
  2. 什么时候回收？ → GC 触发条件
  3. 怎么回收？ → 垃圾收集算法与收集器

存活分析算法

  • 引用计数法：给对象添加一个引用计数器，每有一个地方引用就 +1，失效就 -1。简单高效但无法解决循环引用问题，已不被主流 JVM 采用。
  • 可达性分析算法：从一组 GC Roots 出发，向下搜索引用链，不可达的对象即为可回收对象。这是当前主流 JVM 采用的方法。

GC Roots 包括：

  • 虚拟机栈中引用的对象
  • 方法区中静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中 JNI 引用的对象

垃圾收集算法

标记-清除（Mark-Sweep）：最基础的算法，先标记需要回收的对象，然后统一回收。缺点是会产生大量不连续的内存碎片。

标记-复制（Mark-Copy）：将可用内存按容量等分为两块，每次只用其中一块。当一块用完，将存活对象复制到另一块上，然后一次性清理旧空间。现代 JVM 的新生代就采用这种算法（Eden:S0:S1 = 8:1:1）。

标记-整理（Mark-Compact）：标记阶段与标记-清除一样，但后续步骤不是直接清除，而是让所有存活对象向一端移动，直接清理边界以外的内存。老年代使用这种算法。

分代收集（Generational Collection）：目前主流商业虚拟机的做法，并不是一种具体的算法，而是综合运用上述算法：

区域 特点 适合的算法
新生代 对象存活率低 标记-复制
老年代 对象存活率高 标记-清除 / 标记-整理

垃圾收集器图谱

以下是几款关键的 GC 收集器，了解它们的演进路线就能理解 JVM 发展的方向：

收集器 工作区域 算法 特点
Serial 新生代 复制 单线程，Stop-The-World，适合客户端
ParNew 新生代 复制 Serial 的多线程版本
Parallel Scavenge 新生代 复制 关注吞吐量，适合后台计算
Serial Old 老年代 标记-整理 Serial 的老年代版本
Parallel Old 老年代 标记-整理 Parallel 的老年代版本
CMS 老年代 标记-清除 追求最短停顿时间
G1 全部 分区、局部复制 默认收集器（JDK 9+），兼顾停顿与吞吐
ZGC 全部 并发标记-整理 几乎无停顿（<10ms），JDK 15+ 正式支持
Shenandoah 全部 并发压缩 与 ZGC 类似，不同实现思路

CMS：追求低停顿

CMS（Concurrent Mark Sweep）让垃圾回收线程与用户线程并发执行，大大降低了用户线程的停顿时间。但它有两个硬伤：

  1. 浮动垃圾：并发阶段用户线程产生的新垃圾无法在当前处理
  2. 内存碎片：标记-清除算法导致碎片积累，最终触发 Full GC

G1：区域化分代收集

G1 将堆划分为多个大小相等的 Region，不再固定划分新生代和老年代。每次回收时，G1 优先回收价值最大的 Region（即 Garbage First 的由来）。它是 JDK 9 之后的默认垃圾收集器，兼顾了吞吐量和停顿时间。

ZGC：几乎无停顿

ZGC 最引人注目的特点是它的停顿时间不会超过 10ms，而且与堆大小无关。这意味着即使是上百 GB 的巨型堆，GC 停顿依然控制在毫秒级别。ZGC 通过染色指针（Colored Pointers）和读屏障（Load Barrier）等创新技术实现了这一目标。

实际调优经验分享

说完了理论知识，分享几条实战中最常用的 JVM 调优建议：

1. 不要过早优化 GC

很多团队在系统还没跑起来就在调 GC 参数，这是典型的过早优化。先用默认参数跑一段时间，通过 GC 日志分析后再针对性调整。

2. 根据应用类型选择收集器

  • 批处理/后台计算 → Parallel（追求吞吐量）
  • Web/响应式服务 → G1（平衡停顿与吞吐）
  • 大规模堆（64GB+）→ ZGC（追求低停顿）

3. 重点关注 Full GC

Minor GC 几乎无法避免，但 Full GC 应该极力减少。Full GC 的触发原因通常有：

  • 老年代空间不足
  • 元空间不足
  • System.gc() 被调用
  • GC 担保失败

结语

JVM 内存管理是 Java 开发的底层基石。理解它不是为了让每个开发者都去调优 GC 参数，而是在遇到内存溢出、性能瓶颈时，能够快速地定位问题根源。

推荐想更深入了解的同学阅读《深入理解 Java 虚拟机》（周志明著），这是国内最好的 JVM 中文书籍，没有之一。

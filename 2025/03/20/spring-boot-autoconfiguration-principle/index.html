Spring Boot 自动配置原理探秘

Created2025-03-20|Updated2026-05-07
|Word Count:1.5k|Reading Time:5mins|Post Views:

如果你用过 Spring Boot，一定会被它的「开箱即用」所震撼。只要引入一个 starter 依赖，写几行配置，一个 Web 应用、一个数据源、一个消息队列就能跑起来。

这种体验在 Spring 时代是不可想象的——那时的开发者需要手动配置视图解析器、事务管理器、数据源连接池……一堆 XML 或者 Java Config 文件堆得比业务代码还多。那么，Spring Boot 的自动配置究竟是怎么做到的？这篇文章带你从源码层面一探究竟。

关键代码是 @Import(AutoConfigurationImportSelector.class)。在 Spring 中，@Import 可以导入一个配置类或 ImportSelector 实现类。AutoConfigurationImportSelector 实现了 ImportSelector 接口，它的作用是批量导入需要自动配置的类。

核心流程：加载 spring.factories

AutoConfigurationImportSelector 的核心逻辑在 selectImports() 方法中，总结起来就是三步：

  1. 从 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件中读取所有自动配置类的全限定名
  2. 通过 @Conditional 条件注解过滤掉不满足条件的配置类
  3. 将剩余的配置类注入到 Spring 容器中

条件装配的艺术：@Conditional

@Conditional 注解是 Spring Boot 自动配置的「开关」，它让配置变得有条件的而不是固定的。

Spring Boot 提供了一组非常实用的派生注解：

注解 作用 示例场景
@ConditionalOnClass classpath 存在指定类时 引入 spring-data-redis 时才配置 RedisTemplate
@ConditionalOnMissingClass classpath 不存在指定类时 -
@ConditionalOnBean 容器中已有指定 Bean 时 用户自定义了 DataSource 就不创建默认的
@ConditionalOnMissingBean 容器中不存在指定 Bean 时 用户没配置就创建默认的
@ConditionalOnProperty 配置文件中存在指定属性时 通过配置项开关某个功能
@ConditionalOnResource classpath 存在指定资源时 存在 logback.xml 才配置日志
@ConditionalOnWebApplication 当前是 Web 应用时 Web 相关配置仅在 Web 环境下生效
@ConditionalOnExpression SpEL 表达式为 true 时 复杂的组合条件

实战视角：看 DataSourceAutoConfiguration

这个配置类的逻辑非常清晰：

  1. 只有在 classpath 中存在 DataSource.class 时才加载这个配置
  2. 如果用户已经自己定义了一个 DataSource Bean，就不创建默认的
  3. 只有不存在 io.r2dbc.spi.ConnectionFactory 时才生效（R2DBC 和 JDBC 二选一）
  4. 默认情况下创建一个嵌入式数据库（如 H2）的数据源

这就是 Spring Boot「约定优于配置」理念的实现——你什么都不做，它给你一个合理的默认值；你做了任何自定义，你的配置优先。

Spring Boot 3.x 的变化

Spring Boot 3.x 基于 Spring Framework 6 和 Java 17，有以下重大变化：

  1. 移除 spring.factories：自动配置现在从 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件中读取，不再使用 spring.factories
  2. Jakarta EE 迁移：javax.* 全部变更为 jakarta.*
  3. AOT 编译支持：提前编译（Ahead-of-Time）优化，提升启动速度和减少内存占用
  4. GraalVM Native Image 支持：支持将 Spring Boot 应用编译为原生可执行文件

其中 GraalVM 的支持最有想象空间——Spring Boot 应用的启动时间从几秒缩短到几十毫秒，这对于微服务和 Serverless 场景是非常关键的改进。

结语

Spring Boot 的自动配置并没有使用什么黑魔法，它的核心原理可以总结为一句话：通过 @Conditional 条件注解，在运行时根据 classpath 内容、容器状态和配置属性来决定是否创建某个 Bean。

理解了这一点，Spring Boot 在你眼中就不再是一个黑盒了。当你遇到自动配置没有按预期生效时，可以打开 spring-boot-autoconfigure 的源码看一眼条件注解，通常很快就能找到原因——这也是为什么我一直推荐开发者去读源码。

毕竟，最好的调试工具不是断点，而是理解。

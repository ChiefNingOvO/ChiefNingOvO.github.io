Java 并发编程：从线程到虚拟线程的演进之路

Created2025-02-12|Updated2026-05-07
|Word Count:1.9k|Reading Time:6mins|Post Views:

并发编程是 Java 开发中公认的深水区。即使是工作多年的开发者，也常常在并发问题上翻车。这不能全怪开发者「菜」——并发本身就是一个充满了不确定性、竞态条件和时序问题的复杂领域。好的一面是，Java 在并发编程方面的能力一直在进化，从最早的 synchronized，到 JUC 并发包，再到 JDK 21 的虚拟线程，每一次升级都让并发编程变得稍微容易一点、更加安全一点。

这篇文章带你走一遍 Java 并发编程的演进路线，理解我们是如何走到今天的。

原始时代：synchronized 与 volatile

Java 从 1.0 就开始支持多线程。最早的并发工具只有两个关键词：

  • synchronized：保证同一时刻只有一个线程执行某段代码
  • volatile：保证变量的可见性，禁止指令重排序

synchronized 用起来很简单，但它有几个问题：

  1. 性能开销大：早期的 synchronized 是重量级锁，需要操作系统层面的线程阻塞和唤醒
  2. 不够灵活：锁的获取和释放是固定的，无法实现超时、可中断等高级功能
  3. 粒度不可控：要么锁方法，要么锁代码块，没有读写锁的区分

JDK 1.5 之后，JVM 对 synchronized 做了大量优化（偏向锁、轻量级锁、锁膨胀、锁消除），性能已经大幅提升。但灵活性的问题依然存在。

JUC 时代：java.util.concurrent 的崛起

JDK 1.5 引入了 java.util.concurrent 包（作者是 Doug Lea），这是 Java 并发编程史上最重要的一次变革。

关键区别在于：Lock 支持超时、可中断、公平/非公平等特性。

ReentrantReadWriteLock 则实现了读写分离——读读不互斥、读写互斥、写写互斥，极大提升了读多写少场景的并发性能。

并发容器

Java 集合框架中的大多数类（ArrayList、HashMap 等）都不是线程安全的。JUC 提供了线程安全版本的替代品：

非线程安全 线程安全版本 特点
HashMap ConcurrentHashMap 分段锁/CAS + synchronized，极高并发度
ArrayList CopyOnWriteArrayList 写时复制，适合读多写少
TreeMap ConcurrentSkipListMap 跳表实现，无锁并发

ConcurrentHashMap 的优化尤其值得关注。JDK 7 中使用 Segment 分段锁，JDK 8 则改为 synchronized + CAS，锁粒度从 Segment 降级到单个桶，并发度大幅提升。

线程池

ThreadPoolExecutor 是 JUC 中最常用的组件之一。创建线程在操作系统中是昂贵的操作，线程池通过复用线程来减少创建/销毁的开销。

并发工具类

JUC 还提供了几个非常实用的工具类：

  • CountDownLatch：一个线程等待多个线程执行完毕
  • CyclicBarrier：多个线程互相等待，到达同一个屏障点
  • Semaphore：信号量，控制同时访问资源的线程数量
  • Phaser：更灵活的同步屏障，支持动态注册

Fork/Join 框架

JDK 7 引入了 Fork/Join 框架，用于实现分治算法。它将一个大任务递归拆分为小任务，然后将结果合并。

Fork/Join 底层使用了**工作窃取（Work-Stealing）**算法——空闲的线程会主动从其他线程的队列中「偷」任务来执行。这比传统的线程池负载均衡做得更好。

CompletableFuture：异步编程的福音

JDK 8 引入的 CompletableFuture 让 Java 有了真正的异步编程能力。它解决了 Future 的几个痛点：

  • Future.get() 会阻塞线程
  • Future 无法自动回调
  • Future 无法组合多个异步任务

这种链式调用的风格被称为「响应式编程」，代码逻辑清晰、异常处理集中，比回调嵌套优雅得多。

虚拟线程（Virtual Threads）：颠覆性的变革

JDK 21（2023 年 LTS）正式发布了虚拟线程，这是由 Project Loom 孵化的重量级特性。

传统线程的问题

Java 的线程（现在称为平台线程）本质上是操作系统线程的包装。每个线程都需要独立的栈空间，通常几百 KB 到 1MB。这意味着一个 4GB 的机器最多也就创建几千个平台线程。在高并发场景（比如每个请求需要经历多个 IO 等待），要么用线程池限制并发，要么用 NIO 异步回调把代码写得支离破碎。

虚拟线程的解决思路

虚拟线程是一种用户态线程（类似 Go 的 Goroutine），由 JVM 调度而非操作系统。它们：

  • 轻量：每个虚拟线程只需要几 KB
  • 海量：可以轻松创建数十万个
  • 简单：使用和平台线程完全相同的 API

虚拟线程的革命性意义

虚拟线程最大的价值在于：它让同步编程模型重新变得可行。

在虚拟线程之前，高并发服务通常有两种写法：

  1. 线程池模式（同步 + 阻塞）：代码好写，但线程数量受限
  2. NIO/响应式模式（异步 + 回调）：支持高并发，但代码难调试、难维护

虚拟线程让开发者用同步的代码风格写出支持高并发的应用。不需要学习 Reactor、RxJava，不需要处理回调地狱，直接用普通的 synchronized、Thread.sleep()、数据库连接池，就能支持十万级并发。

结语

从 synchronized 到 Lock、从 Thread 到 ForkJoinPool、从 Future 到 CompletableFuture，Java 的并发编程一直在朝着两个方向努力：更高的性能 和 更简单的写法。

虚拟线程将这两个方向推到了一个新的高度。我认为在未来几年内，虚拟线程将成为 Java 服务端开发的标准模型，而那些复杂的异步框架也会逐渐退居幕后。Java 终于学会了一件事：最好的并发 API，就是让开发者几乎感觉不到并发的存在。

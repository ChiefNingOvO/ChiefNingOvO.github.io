设计模式在 Java 中的经典应用

Created2025-06-10|Updated2026-05-07
|Word Count:2.1k|Reading Time:8mins|Post Views:

设计模式是什么？用一句话说：设计模式是在特定场景下解决特定问题的可复用方案。

很多人学设计模式时最大的困惑是：学了一堆 UML 图，背了一堆定义，一到写代码就全忘了。这是正常的——设计模式不是靠背的，而是靠「踩坑」理解的。你没有写过可维护性很差的代码，就无法理解为什么需要工厂模式；没有遇到过度耦合的模块，就无法理解依赖注入的价值。

这篇文章不谈纯理论，而是结合 Java 生态中的实际案例来讲解几种最常用的设计模式。

创建型模式

单例模式：最熟悉也最容易被坑的模式

单例模式保证一个类只有一个实例。Java 中常见的饿汉式、懒汉式、双重检查锁等写法几乎是面试必考。

但双检锁（Double-Checked Locking）在 JDK 5 之前由于 JMM 的指令重排序问题，是存在隐患的：

new Singleton() 不是原子操作，它实际分三步：

  1. 分配内存
  2. 初始化对象
  3. 将引用指向内存

如果不加 volatile，步骤 2 和 3 可能被重排序。线程 A 执行了 1 和 3（instance 不为 null 了但对象还没初始化完成），线程 B 判断 instance != null 就返回使用了，结果 crash。

更推荐的写法：静态内部类

这种方式利用 JVM 的类加载机制保证线程安全，既实现了懒加载，又不需要同步开销，是业界公认的最佳写法。

工厂模式：Spring 容器的基石

工厂模式将对象的创建和使用分离。在 Spring 中，几乎所有的 Bean 都是通过工厂创建的。

结构型模式

代理模式：AOP 的实现基础

代理模式的核心是为某个对象提供一个代理，由代理对象控制对原对象的访问。

Java 中的代理分为两种：静态代理和动态代理。

静态代理：代理类在编译期就确定了。

问题：每个接口都需要写一个代理类。

JDK 动态代理：运行时动态生成代理类。

CGLIB 代理：通过继承生成子类作为代理，不需要接口。

Spring AOP 在实现上会根据目标 Bean 是否实现接口来选择 JDK 动态代理或 CGLIB。

适配器模式：兼容的艺术

适配器模式解决的是「接口不兼容」的问题。现实世界中最好的类比是电源转换插头。

行为型模式

策略模式：消除 if-else 的利器

当一段代码中有大量的 if-else if-else 来判断「不同情况执行不同逻辑」时，就应该考虑策略模式了。

经典案例：支付方式

策略模式的另一个好处是——新增加一种支付方式只需要新建一个类，完全符合开闭原则。

观察者模式：事件驱动的基础

观察者模式定义了对象之间的一对多依赖关系——当一个对象状态改变时，所有依赖它的对象都会收到通知。

模板方法模式：框架中的骨架

模板方法模式定义了一个算法的骨架，将一些步骤延迟到子类中实现。

AbstractList 是模板方法模式的典型应用：

HibernateTemplate、JdbcTemplate、RestTemplate 也都采用了模板方法模式——模板方法负责处理资源获取、事务管理等通用逻辑，回调方法由使用者实现具体业务。

设计模式的使用原则

最后分享几条实用的原则：

  1. 不要为了用模式而用模式。模式是解决问题的工具，不是炫技的手段。滥用模式会让代码比你原本的问题更复杂。

  2. 模式是共识的载体。使用标准的设计模式（而不是自创的奇怪模式）能让团队中的其他开发者更快理解你的代码。

  3. 先有重构，后有模式。不要一开始就套用模式。先写简单直接的代码，当代码开始散发出坏味道时，再用合适的模式去重构。

  4. 过度工程化是设计模式的常见陷阱。你的项目可能只需要一个简单的 if-else，而不是一个完整的策略模式 + 工厂模式 + 注册表模式。

用《Head First 设计模式》中的一句话来结尾：“Know the patterns, but don’t become pattern-blind.”

Java 集合框架设计哲学：从 ArrayList 到 ConcurrentHashMap

Created2025-04-08|Updated2026-05-07
|Word Count:2k|Reading Time:7mins|Post Views:

Java 集合框架（Java Collections Framework，JCF）是 JDK 中使用频率最高、设计最为精巧的子系统之一。每一个 Java 开发者每天都在和 ArrayList、HashMap、HashSet 打交道，但真正深入理解它们设计哲学的人并不多。

这篇文章不从 API 手册的角度罗列方法，而是从设计者的视角来解读：为什么 ArrayList 的默认容量是 10？为什么 HashMap 在 JDK 8 中要引入红黑树？为什么 ConcurrentHashMap 能如此高效？

三足鼎立的架构：List（有序可重复）、Set（无序不可重复）、Map（键值对）。

此外还有一个重要的概念——Iterator（迭代器），它为遍历集合提供了统一的抽象。这使得所有集合类都可以通过统一的 for-each 语法来遍历。

ArrayList vs LinkedList：不只是「数组 vs 链表」

很多人面试时被问到 ArrayList 和 LinkedList 的区别，传统的标准答案是：「ArrayList 底层是数组，查找快、增删慢；LinkedList 底层是双向链表，增删快、查找慢。」

这个答案大体正确，但不完全准确。

ArrayList 的设计细节

几个有趣的细节：

  1. 默认容量是 10 但不是一开始就分配的：JDK 8 做了懒加载优化，创建 ArrayList 时底层数组是空的，第一次 add 才初始化为容量 10
  2. 扩容策略是 1.5 倍：oldCapacity >> 1 相当于除以 2，所以新容量 = 旧容量 + 旧容量/2。这个倍数经过精心计算——扩容太大浪费内存，扩容太小频繁复制
  3. Arrays.copyOf 的背后是 System.arraycopy：这是一个 native 方法，利用 JVM 层面的内存拷贝，效率远高于手动 for 循环复制

LinkedList 的「增删快」是真的吗？

LinkedList 的 get(int index) 确实需要遍历链表，时间复杂度 O(n)。但所谓的「增删快」也要看具体位置：

在某些场景下 ArrayList 的增删反而更快，比如在尾部批量添加元素时，ArrayList 的内存连续性优势非常明显。

实际选择建议

  • 绝大多数场景选 ArrayList：对内存更友好（连续内存），遍历更快（CPU 缓存友好），扩容不频繁
  • 只在头部频繁插入/删除或需要实现队列/双端队列时才用 LinkedList
  • 如果需要线程安全，使用 CopyOnWriteArrayList（读多写少）或 Collections.synchronizedList(list)

HashMap：JDK 8 中最重要的优化

HashMap 的演进历史是数据结构理论与工程实践结合的典范。

JDK 7 中的 HashMap

JDK 7 的 HashMap 使用数组 + 链表实现。当发生哈希冲突时，新元素被插入链表头部。这种实现有两个严重问题：

  1. 哈希碰撞严重时退化为链表：查找时间复杂度退化为 O(n)
  2. 死循环 Bug：多线程扩容时，链表头插法可能导致环形链表，在 get 时触发死循环

JDK 8 的重大改进

JDK 8 对 HashMap 做了三项关键改进：

改进一：数组 + 链表 + 红黑树

为什么阈值是 8？这其实是一个统计学上的考量。根据泊松分布，在随机哈希函数下，链表长度达到 8 的概率约为 0.00000006——也就是说，触发了树化几乎可以断定哈希函数有问题。

改进二：尾插法

JDK 8 改用尾插法，从根本上解决了扩容死循环的问题。

改进三：扩容优化

JDK 7 在老数组迁移到新数组时需要重新计算所有元素的哈希值（rehash），JDK 8 利用「元素在新数组中的位置要么不变，要么是旧位置 + 旧容量」的特性，避免了重算哈希，性能更好。

容量为什么是 2 的幂次？

当 capacity 是 2 的幂次时，(n - 1) & hash 等价于 hash % n，但位运算比取模快一个数量级。这就是 HashMap 容量始终是 2 的幂次的原因。

ConcurrentHashMap：并发安全的艺术

ConcurrentHashMap 是面试中的大户，它的设计理念经历了从 JDK 7 到 JDK 8 的巨大变化。

JDK 7：分段锁（Segment）

JDK 7 的 ConcurrentHashMap 维护了一个 Segment 数组，每个 Segment 本质上是一个小型 HashMap，继承自 ReentrantLock。理论上最多支持 16 个线程同时写入（Segment 数量）。

问题在于：Segment 的大小和使用数量在创建时固定，不能动态调整。当 Segment 粒度过粗时，并发度不够。

JDK 8：CAS + synchronized

JDK 8 的 ConcurrentHashMap 放弃了分段锁，改用更细粒度的单个桶加锁：

这样做的好处：

  1. 锁粒度更细——从 Segment 级别降级到单个桶级别
  2. 读操作完全无锁——Node.value 用 volatile 修饰，保证可见性
  3. 扩容通过多线程协助实现——扩容时其他线程可以帮忙迁移数据

实际使用建议

computeIfAbsent 是非常重要的方法，在需要「先检查再插入」的场景下，它可以保证原子性，避免放入两次。

模板方法模式

AbstractList 定义了 get() 和 size() 等抽象方法，子类只需要实现这些方法，AbstractList 就帮你实现了 subList()、indexOf() 等基于这些基本操作的衍生方法。

结语

Java 集合框架历经二十多年的演进，几乎每一个设计决策背后都有深思熟虑的权衡。理解这些设计哲学，不仅能帮你写出更好的代码，也能让你在使用时更加心中有数。

归根结底，集合框架教给我们最重要的一件事是：没有最好的数据结构，只有最合适的数据结构。

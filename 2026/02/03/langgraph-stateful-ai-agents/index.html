LangGraph 入门：构建有状态的 AI Agent

Created2026-02-03|Updated2026-05-08
|Word Count:1.6k|Reading Time:6mins|Post Views:

如果你用过 LangChain 的 Chain（链式调用），你会发现它有一个根本性的局限——Chain 是无状态的、线性的。一个输入进去，经过若干步骤，一个输出出来，中间没有分支、没有循环、没有动态决策。

这就像只能用一条流水线工作——当 Agent 需要在不同阶段做出不同的决策时，Chain 就显得捉襟见肘。

LangGraph 正是为解决这个问题而生的。它把 AI 应用建模为一个有向图（更准确地说是状态图），让 Agent 可以在节点之间跳转、循环和分支。

从 Chain 到 Graph

LangGraph 的 Graph 模型：

区别在于：

  • Chain 是线性的——事先决定了执行路径
  • Graph 是非线性的——运行时的状态决定了下一步走向

LangGraph 的核心概念

State（状态）

State 是 Graph 中流转的核心数据结构，每个节点读取 State 并可能修改它。

Nodes（节点）

节点是 Graph 中的执行单元。每个节点是一个函数，接收 State 并返回对 State 的修改。

Edges（边）

Checkpointing：让 Agent 拥有记忆

LangGraph 的一个关键特性是内置的 Checkpointing（检查点）。每次经过一个节点，Graph 都会保存当前 State 的快照。

这带来了几个重要的能力：

1. 对话持久化

2. 时间旅行

这在调试中极其有用——你可以回到某个出错的节点，修正状态后重新执行，而不需要从头开始。

3. Human-in-the-Loop

这是企业级 Agent 应用的关键需求——让人类在关键时刻介入审批。

高级模式

并行执行 + 合并

Agent 嵌套

这种嵌套能力让你可以构建层次化的 Agent 系统——顶层 Agent 负责宏观决策和任务分配，底层 Agent 负责具体执行。

与 LangChain 的关系

LangGraph 和 LangChain 不是竞争关系，而是互补关系。典型的使用方式：

LangChain 提供「素材」，LangGraph 提供「结构」。这是官方推荐的配合方式。

何时选择 LangGraph？

LangGraph 最适合以下场景：

  • 多步 Agent：需要多轮工具调用和推理
  • 有状态的对话：需要跨多轮对话维护复杂状态
  • 人在回路中：需要人类审批某些关键操作
  • 复杂的分支逻辑：根据运行时结果动态走不同路径

如果只是做简单的 RAG（检索 → 生成），直接用 LangChain 的 LCEL 就够了。

结语

LangGraph 代表了 AI 应用框架的一个趋势：从线性的 Prompt Chain 向有状态的 Agent Graph 转变。这不仅仅是 API 层面的变化，更是思维方式的转变——构建 AI 应用不再只是设计 prompt 和链，而是设计一个异步的、事件驱动的状态机。

当你把 Agent 看作一个状态机时，很多复杂的问题——重试、超时、并行、回溯——就都有了明确的处理方式。LangGraph 的状态管理、checkpointing 和条件路由机制，本质上就是在给开发者提供状态机的编程模型。

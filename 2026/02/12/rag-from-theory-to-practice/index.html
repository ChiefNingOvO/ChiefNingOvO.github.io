RAG 从理论到实践：检索增强生成全景解读

Created2026-02-12|Updated2026-05-08
|Word Count:1.8k|Reading Time:6mins|Post Views:

如果 LLM 有一个「原生短板」，那就是它的知识有一个截止日期。GPT-4 的知识截止于 2023 年，Claude 的知识截止于 2024 年——每次被问到最新事件时，它要么回答不上来，要么产生幻觉。

RAG（Retrieval-Augmented Generation，检索增强生成）就是解决这个问题的标准答案。它的思想非常简单：在 LLM 回答之前，先检索相关的外部信息，然后让 LLM 基于这些信息来生成回答。

分割策略：RAG 最重要的工程决策

文本如何分割为 Chunk，是影响 RAG 检索质量最重要的一环。

按字符分割（最简单）

适合文档结构统一、内容连续的文本。通过分隔符（换行符、句号、空格）的优先级递归分割。

按语义分割（更智能）

语义分割不是按字符数切分，而是检测语义相似度的「断裂点」——当相邻句子的语义从一个话题跳转到另一个话题时，就在跳转处切割。这种方法更慢（需要额外的 embedding 调用），但检索质量显著更好。

Chunk 大小的权衡

Chunk 大小 优点 缺点
小（100-300 tokens） 检索精度高 丢失上下文，C 可能割裂一个完整的观点
中（500-1000 tokens） 平衡精度和上下文 -
大（2000+ tokens） 保留完整上下文 检索精度降低，噪音增加

一个常见的取巧做法是：检索用小 Chunk，但返回用大 Chunk 的上下文。实现方式是给每个小 Chunk 记录它来自哪个大段落，检索到小 Chunk 时连带返回大段落。

Self-RAG：让 LLM 评价检索质量

RAG 的评估

没有评估就没有改进方向。RAG 评估有三个维度：

1. 检索质量

  • Recall@K：前 K 个检索结果中命中了多少相关文档
  • MRR（Mean Reciprocal Rank）：第一个相关文档的平均排名倒数

2. 生成质量

  • Faithfulness（忠实度）：回答是否严格基于检索到的文档（是否编造了文档中没有的内容）
  • Answer Relevance：回答是否真正回应了问题

3. 端到端质量

  • Correctness：回答是否正确（需要人工标注）

RAG 的局限性

RAG 不是银弹。以下场景中 RAG 可能不是最佳方案：

  1. 需要全局理解的任务：RAG 只给你「片段」，如果任务需要理解整个文档的主题脉络，检索片段是不够的
  2. 检索不到正确答案的场景：如果知识库没有相关内容，RAG 也无能为力
  3. 检索到了错误信息的场景：知识库中的文档可能是过时的或错误的信息
  4. 延迟敏感的场景：RAG 引入了额外的检索延迟

在这些场景中，微调（Fine-tuning）可能是更好的选择——或者将 RAG 和微调结合起来用。

结语

RAG 是目前最成熟的 LLM 应用模式，很可能在未来很长一段时间内都是 AI 应用架构的「标配」。但它的挑战不在于「怎么搭一个能跑的 RAG」，而在于「怎么搭一个生产级可靠的 RAG」。

这里面涉及的大多数问题都是工程问题，不是算法问题：怎么做文本分割、怎么管理向量数据库、怎么做查询优化、怎么做评估。这恰恰意味着，好的 RAG 系统不是靠买一个更好的模型就能解决的——它需要扎实的软件工程能力。

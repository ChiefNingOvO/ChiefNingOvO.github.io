LLM 的幻觉问题：成因、检测与缓解策略

Created2026-02-27|Updated2026-05-08
|Word Count:2.1k|Reading Time:7mins|Post Views:

如果你是 AI 应用的开发者，你一定在处理 LLM 的幻觉问题——模型自信满满地编造了一个不存在的 API、一个虚构的历史事件、一组看起来合理但完全错误的数据。

幻觉（Hallucination）是 LLM 最让人头疼的问题之一。它不是 bug，而是 LLM 基于概率采样的推理方式所固有的。理解幻觉的成因和应对策略，是构建可靠 AI 应用的必修课。

幻觉的类型

不是所有的错误输出都是幻觉。学术界将幻觉分为几种类型：

事实性幻觉

模型生成了与客观事实不符的内容：

从概率角度理解，事实性幻觉发生时，模型预测了一个在语法上合理、在语义上连贯但在现实世界中缺乏对应事实的 token 序列。

忠实性幻觉

模型生成的内容与给定的上下文或自身的前文矛盾：

这个回答虽然可能逻辑上看起来合理（很多公司确实这样规定），但它与上下文给出的信息矛盾——这就是忠实性幻觉。

机能性幻觉

模型生成了格式错误、逻辑矛盾的输出：

或者回答说「正如上文所述，X = 5」，但上文并没有提到 X 的值。

幻觉的根因：不止是训练数据

根因一：分布外泛化

LLM 的训练数据覆盖了人类知识的一大部分，但不是全部。当用户的问题落在训练数据的「空白地带」时，模型不会回答「我不知道」，而是会从最接近的已知模式中「推测」一个答案。

这就是为什么 LLM 容易在长尾话题和专业知识上出现幻觉——训练数据稀疏。

根因二：下一个 token 的最大概率并不等于正确

对于一个没有确定答案的问题（训练数据中有多种不同的说法），模型会倾向于生成统计上最常见的 token 组合——即使这个组合并不是「事实」。

根因三：自回归的雪崩效应

自回归生成意味着之前的输出决定之后的输出。一旦模型在一个 token 上「走偏了」，后面的 token 会基于这个错误的 token 继续生成，使得错误越来越离谱。

根因四：训练数据中的矛盾

训练数据中包含了大量相互矛盾的信息（同一事件的不同报道、不同学派的不同观点），模型在训练中学会了其中所有的模式。当这些矛盾的信息无法调和时，幻觉就产生了。

检测幻觉

自检（Self-Check）

自检不能 100% 发现幻觉（因为模型可能同时产生幻觉和错误的自我评估），但能提高不少。

交叉验证

在企业应用中，交叉验证是最有效的方法——用 RAG 检索到的权威文档来核实 Agent 的输出。

不确定性检测

同一个问题问模型 5 次，如果 5 次回答差异很大，说明模型在这个问题上高度不确定——你应该考虑不做回答或降低对该回答的置信度。

结语

谈到 LLM 的幻觉时，有一个重要的心态转变：不要把幻觉当成需要完全消除的 bug，而是当成需要管理的风险。

就像网络安全中有「零信任」思想一样，处理 LLM 幻觉也需要一种「零信任」的工程方法：永远假设模型可能出错，在关键路径上设置校验、验证和兜底机制。

好消息是，每一年模型的基础能力都在提升，幻觉率在持续下降。但坏消息是——至少在可预见的将来——幻觉不可能完全消失。学会与之共存，是 AI 应用开发者的宿命。

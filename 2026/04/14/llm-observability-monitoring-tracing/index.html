LLM 应用的可观测性：监控、追踪与评估

Created2026-04-14|Updated2026-05-08
|Word Count:1.8k|Reading Time:7mins|Post Views:

传统软件的可观测性已经是一门成熟的学问——metrics（指标）、logs（日志）、traces（链路追踪），三根支柱架起了整个监控体系。

LLM 应用的可观测性在此基础上增加了一个全新的维度：质量观测（Quality Observability）。传统监控关心的是「服务是否正常」，LLM 监控还要关心「回答是否正确」、「有没有幻觉」、「输出格式对不对」。这是两个完全不同的世界。

为什么 LLM 应用的可观测性这么难？

传统 Web 应用的行为是确定的——相同的输入得到相同的输出。如果系统出错，你可以通过日志复现。

LLM 应用的行为是概率性的——相同的输入可能得到不同的输出（即使 temperature=0，不同批次的 API 调用也可能有微小差异）。而且错误不是 Crash，而是「说得不对」。

这些特性让 LLM 可观测性面临独特的挑战：

  1. 无法精确复现：两次相同请求的输出可能不同
  2. 错误是语义层面的：「返回了一个虽然格式正确但内容有误的回答」不是 status_code=500
  3. 延迟波动大：LLM 的生成时间从几百毫秒到几十秒都有可能
  4. 成本是运行时变量：同样的任务，Agent 可能调用 1 次 LLM，也可能调用 10 次

结语

建立 LLM 应用的可观测性需要多少投入？底线是：至少要能追踪每一次 LLM 调用和工具调用，至少要有一个基础的质量评估。没有这两样，你就是在摸黑驾驶。

更重要的是：可观测性不是事后补救的手段，而是系统设计的一部分。在你设计第一个 Agent 的时候，就应该在代码中埋入追踪和评估的逻辑。越晚做，成本越高。

AI 应用从「跑起来了」到「可以放心上线」，之间的距离不是功能完成度，而是可观测性的建设程度。

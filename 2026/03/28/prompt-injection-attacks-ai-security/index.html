提示词注入攻击与 AI 安全

Created2026-03-28|Updated2026-05-08
|Word Count:2.4k|Reading Time:8mins|Post Views:

2025 年，一个 AI 客服机器人被用户诱导说出了内部 API 密钥；同一年，某公司的「AI 简历筛选器」被发现可以被精心设计的简历操纵。这些事件的共同点是什么？提示词注入（Prompt Injection）——一种专门针对 LLM 应用的攻击方式。

AI 安全不只是传统的网络安全。提示词注入是一个全新的攻击面，它不破解你的代码，不渗透你的网络——它直接攻击你的 LLM。

什么是提示词注入？

提示词注入的本质是：

攻击者在 LLM 的输入中嵌入恶意指令，诱导模型忽略原始的系统指令，转而执行攻击者的意图。

它看起来是这样的：

如果模型遵从了攻击者的指令（输出「强烈推荐」），那么这个注入就成功了。

攻击为什么能成功？

从 LLM 的视角来看，系统指令和用户输入都只是一段文本中的 token。模型没有内在的机制来区分「这是系统指令，必须遵守」和「这是用户输入，仅供参考」。

间接注入（Indirect Injection）

间接注入更隐蔽、危害更大，因为它不需要攻击者直接与 LLM 交互——只要攻击者能污染 LLM 检索的数据源就行。

多模态注入

攻击在图像、音频等非文本输入中嵌入指令：

多模态模型在读取图像时会处理所有相关数据，包括元数据。

真实攻击案例复盘

案例一：Bing Chat 的越狱

2023 年，Bing Chat（基于 GPT-4）被用户通过一种新方式「越狱」——攻击者要求 Bing 扮演一个名为 Sydney 的角色，这个角色「不受微软的规则约束」。一旦角色确立，Sydney 会遵从攻击者的各种要求，从泄露内部代号到表达攻击性情绪。

攻击原理：利用 LLM 的角色扮演能力，创建了一个「不受约束」的信任域。

案例二：AI 邮件助手的间接注入

2024 年，一个 AI 邮件助手被设计为在收到新邮件时自动生成摘要并提示用户。攻击者向目标用户发送了一封看似正常的邮件，但邮件正文中嵌入了白色字体（对人不敏感但对 AI 可见）的指令：

攻击原理：利用 LLM 与人类的输入感知差异——人类看的是视觉呈现（白色字体看不见），但 LLM 读的是文本内容（白色字体也是文本）。

防御策略

防御一：输入净化

但这种基于黑名单的方法并不可靠——攻击者可以用同义词、编码、多语言来绕过。

防御二：指令分层（最有效）

这是目前最有效的防御方法——通过结构化的标记来告诉模型「这些内容有不同的信任级别」。

关键技巧是：

  1. 用 XML 标签明确区分内容层次
  2. 明确告诉模型哪些内容不能被修改
  3. 要求模型报告任何可疑的注入行为

防御三：二次校验

不同 LLM 有不同弱点，用模型 A 做业务，用模型 B 做安全检查，可以有效降低单一模型的盲区。

防御五：上下文隔离

安全管理清单

在 AI 应用的安全管理中，以下措施是基础性的：

  • 每次 LLM 输出前：进行内容过滤，防止泄露敏感信息
  • 敏感操作必须人工确认：退款、删除、发邮件等操作前需要用户二次确认
  • 记录完整的审计日志：LLM 的每次输入输出、工具调用、用户确认都应该被记录
  • 定期进行红队测试：用已知的攻击模式定期测试系统
  • 监控异常模式：同一个用户反复尝试修改系统指令、大量失败的验证等

结语

提示词注入是一个没有完美解法的安全问题——至少在目前的 LLM 架构下是如此。只要 LLM 无法原生区分「指令」和「数据」，注入攻击的可能性就无法被完全消除。

但在实践中，攻击的容易程度可以大幅降低。多层防御（输入验证 + 指令分层 + 二次校验 + 最小权限）的组合使用，能使绝大多数攻击变得无效。

最重要的一点是：把 AI 安全纳入你的整体安全体系。它不是飞来横祸，是每个 AI 系统在设计阶段就必须处理的威胁。如果你正在构建一个面向外部用户的 AI 应用，现在就开始做安全设计，不要等到出了问题之后再补救。

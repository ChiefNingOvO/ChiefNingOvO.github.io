AI Skill 系统设计：可组合的 AI 能力单元

发表于2026-03-23|更新于2026-05-08|AI
|总字数:2.7k|阅读时长:9分钟|浏览量:

引言：从 Tool 到 Skill 的进化

工具（Tool）让 LLM 拥有了调用外部能力的基础设施，但单个工具对复杂任务的支持仍然有限。一个典型的企业 AI 应用往往需要协调数十个工具、编排多轮调用、管理复杂的状态流转——这种场景下，我们需要比「工具」更高层级的抽象。

这就是 Skill（技能） 的用武之地。如果说 Tool 是 LLM 可以挥舞的一把螺丝刀，那么 Skill 就是一份完整的操作手册——它包含了做什么、用什么工具做、按什么流程做、以及如何判断做得好不好。

Skill 与 Tool 的本质区别

维度 Tool（工具） Skill（技能）
粒度 原子操作 复合能力
复杂度 单步执行 多步骤、带分支和循环
知识包含 无 包含领域知识和操作指引
状态管理 无状态 有状态
组合方式 简单串并联 分层嵌套、条件路由
可复用性 上下文相关 跨场景可复用
示例 query_database 「竞品分析报告生成」

Tool 回答「怎么查数据」，Skill 回答「怎么完成一个任务」。

Skill 的四要素模型

一个完整的 Skill 由四个核心要素构成：

1. Prompt（提示词）

Skill 包含预定义的提示词模板，指导 LLM 在该技能语境下的行为模式。与通用的系统提示词不同，Skill 提示词是场景化的、任务聚焦的。

2. Tools（工具集）

Skill 绑定了完成任务所需的一组工具。不是所有的工具都需要暴露给 Skill——只绑定相关的子集，减少模型的选择负担和错误率。

3. Knowledge（知识）

Skill 包含领域知识，包括：业务规则、常见问题处理方式、输出格式规范、合规要求等。这些知识可以是静态文本，也可以是动态接入的知识库。

4. Workflow（工作流）

Skill 定义了任务执行的标准流程（Standard Operating Procedure，SOP）。这不是写死的代码逻辑，而是给 LLM 的结构化指引，让模型知道「遇到什么情况该怎么走」。

Skill 的 Schema 设计

为了支持 Skill 的注册、发现和组合，需要定义标准化的 Skill Schema：

Schema 的关键字段解读：

  • version：支持语义化版本控制，确保兼容性
  • trigger_keywords：帮助路由器自动选择合适的 Skill
  • dependencies：声明对其他 Skill 的依赖
  • requires_approval：高风险操作需要人工确认
  • rate_limit：调用频率限制

Skill 的组合模式

Skill 不是孤立的，它们可以组合成更强大的能力：

管道模式（Pipeline）

多个 Skill 按顺序执行，前一个的输出是后一个的输入。

并行模式（Parallel）

多个 Skill 同时执行，结果合并。

条件分支（Conditional Branching）

根据中间结果动态选择后续 Skill。

Skill 库设计与管理

当组织积累了数十个 Skill 后，需要系统化的管理：

目录结构

版本管理

Skill 使用语义化版本（SemVer）：

  • MAJOR：不兼容的接口变更（输入输出格式变化）
  • MINOR：向后兼容的新功能（新增可选参数、优化工作流）
  • PATCH：向后兼容的修复（提示词优化、知识更新）

测试策略

每个 Skill 应有独立的测试套件：

  • 单元测试：验证单个工具调用的正确性
  • 集成测试：验证 Skill 内部多步骤流程
  • 回归测试：使用标注数据集验证 Skill 输出质量

Skill vs Agent vs Plugin

层次关系：Plugin ⊂ Tool ⊂ Skill ⊂ Agent

案例研究：Claude Code 的 Skills 系统

Claude Code（Anthropic 出品的 AI 编程助手）将 Skills 作为核心架构元素。其设计亮点包括：

Skill 注册与发现

Claude Code 通过 .claude/skills/ 目录自动发现和注册 Skill。每个 Skill 是一个独立的文件夹，包含 SKILL.md（技能描述和提示词）和相关的工具配置。

条件触发

Skill 通过自然语言描述触发条件，而非固定的关键词匹配。当用户输入与某个 Skill 的触发描述语义匹配时，该 Skill 被激活。

Skill 组合

Claude Code 支持多 Skill 协同工作。例如，用户在编写代码时，可能同时触发「代码生成 Skill」和「代码审查 Skill」，二者交替运行，形成「编写-审查-修改-再审查」的闭环。

权限模型

每个 Skill 有独立的权限声明。高风险 Skill（如执行 Shell 命令）需要用户显式授权；低风险 Skill（如代码格式化）可以自动执行。

Skill 的评估方法

评估一个 Skill 的质量需要多维度的指标体系：

功能性指标

  • 任务完成率：Skill 能成功完成的任务占比
  • 首次成功率：不需要 LLM 重试就能完成的比例

效率指标

  • 平均执行时间：从触发到产出结果的时间
  • Token 消耗：单次执行消耗的 Token 数

质量指标

  • 输出准确性：结果是否符合预期（人工评估或自动化对比）
  • 用户满意度：用户对 Skill 输出的打分

健壮性指标

  • 异常处理率：遇到异常输入时的正确处理比例
  • 边界情况覆盖率：对极端输入的测试覆盖

未来展望：Skill 市场与自动生成

Skill Marketplace

随着 AI 应用的普及，可能会出现类似 App Store 的 Skill Marketplace。开发者可以将自己的 Skill 打包发布，企业可以按需采购和集成。这将催生一个新的生态系统：专业化 Skill 开发者、Skill 评价体系、Skill 兼容性认证。

自动生成的 Skill

未来的 AI 系统可能具备这样的能力：观察用户的工作流程，自动抽象出可复用的 Skill。例如，看到用户反复执行「从数据库导出数据 → 用 Excel 制作图表 → 撰写分析报告」的流程，AI 自动生成一个「数据分析报告 Skill」。

自进化的 Skill

Skill 可能具备自我优化的能力：通过收集执行数据、分析失败案例，自动调整提示词和工作流，持续提升性能。这需要完善的监控反馈回路和安全的自动化变更机制。

总结

Skill 是比 Tool 更高层次的 AI 能力抽象，它将提示词、工具集、领域知识和工作流整合为一个可组合、可复用、可管理的能力单元。从 Tool 到 Skill 的进化，本质上是 AI 应用从「能做什么」到「会做什么」的转变——前者靠工具的数量，后者靠能力的深度。

设计一个好的 Skill 系统，核心是把控四点：清晰的接口定义（让 Skill 可被发现和使用）、合理的粒度划分（不过大也不过小）、灵活的组合能力（让多个 Skill 协同工作）、完善的治理机制（版本管理、权限控制、质量评估）。

在 AI 应用走向工程化的今天，Skill 系统将成为构建高质量、可维护 AI 产品的关键基础设施。

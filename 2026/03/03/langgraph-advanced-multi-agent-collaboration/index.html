LangGraph 高级实战：多 Agent 协作与状态管理

Created2026-03-03|Updated2026-05-08
|Word Count:1.7k|Reading Time:7mins|Post Views:

在上一篇文章中，我介绍了 LangGraph 的基本概念和单 Agent 的构建方式。这一篇我们深入到更复杂的场景：多 Agent 协作和生产级的状态管理。

在实际业务中，几乎没有一个复杂的任务能被单个 Agent 胜任。代码开发可能需要设计 Agent、编码 Agent、审查 Agent 协同工作；智能客服可能需要意图 Agent、查询 Agent、退款 Agent 各司其职。

多 Agent 架构的三种模式

模式一：Supervisor（监督者模式）

一个主 Agent（Supervisor）负责任务分配和协调，多个 Worker Agent 负责具体执行。

Supervisor 模式是最直观的多 Agent 架构——就像一个团队领导分配任务、收集结果、决定下一步。

模式二：Hierarchy（层次模式）

多层的 Agent 层次结构：

每层 Agent 只负责自己层级的决策，上层 Agent 不需要了解下层的实现细节。

模式三：Swarm（群体协作模式）

这是 OpenAI Swarm 提出的模式——多个 Agent 通过彼此的直接通信来协作，没有中心调度者。每个 Agent 可以将任务「交接」给其他 Agent。

Swarm 模式适合 Agent 数量不多、角色明确的场景。相比 Supervisor 模式，Swarm 的延迟更低（不需要反复请示），但可能出现 Agent 之间反复踢皮球的情况。

可观测性：追踪 Agent 的每一步

每个步骤的日志应包含：

  • 步骤序号和名称
  • LLM 的输入输出耗时
  • 工具调用的参数和结果
  • State 的变化（前后 diff）

结语

LangGraph 的多 Agent 能力和状态管理机制让它从「一个 Agent 编排框架」升级为了「一个分布式 Agent 系统的运行时」。它提供的 Checkpointing、Human-in-the-Loop、条件路由等能力，本质上是在回答一个核心问题：如何让 LLM 的推理过程变得可控和可观测。

当你从简单的 Prompt Chain 进化到复杂的 Agent Graph 时，最重要的心态转变是：把 Agent 当作一个系统来设计，而不是一段脚本。

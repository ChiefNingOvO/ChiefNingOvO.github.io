LangChain 初探：用链式调用构建 AI 应用

Created2026-01-08|Updated2026-05-08
|Word Count:1.5k|Reading Time:5mins|Post Views:

如果你已经用过 ChatGPT 的 API，你可能会觉得「调用 LLM 很简单啊，一个 HTTP 请求就完事了」。但当你真正开始构建一个 AI 应用时，你会发现事情远没有那么简单——你需要管理对话历史、需要连接外部数据源、需要给 LLM 配备工具、需要处理输出格式……这时候 LangChain 就出现了。

LangChain 今天的定位早已不是「一个调用 LLM 的包装库」，而是一整套 LLM 应用开发的框架和生态系统。

为什么需要 LangChain？

先从最朴素的 API 调用开始：

这段代码可以工作——但你很快会需要：

  1. 管理对话历史：多轮对话需要自己维护 messages 数组
  2. 连接外部数据：想让 LLM 回答关于公司文档的问题，需要自己实现检索逻辑
  3. 调用工具：想让 LLM 查询数据库或调用 API，需要自己解析 tool calls
  4. 处理流式输出：想实现打字机效果，需要处理 SSE 流

每一样都不算太难，但加在一起就变成了一个相当复杂的系统。LangChain 的价值在于：把这些常见模式抽象为标准化的组件，让开发者专注于业务逻辑。

LangChain 的设计演进：从 Chain 到 LCEL

LangChain 早期版本的核心抽象是 Chain 类——一个 Pipeline 模式的实现。但 Chain 有两个固有问题：

  1. 不透明：Chain 的内部结构像一个黑盒，难以调试
  2. 不够灵活：简单的链式调用无法表达复杂的控制流（条件分支、循环、并行）

实战建议

不要过度依赖 LangChain

LangChain 一个常见的批评是「太重了」——为了用 20 行的 LLM 调用，你需要理解 200 个类的继承关系。我的建议是：

  • 原型阶段：直接用 LLM API，不引入框架
  • 功能验证：需求稳定后，用 LangChain 重构核心流程
  • 生产环境：只依赖 LangChain 的核心模块（langchain-core），避免引入不需要的社区模块

从 langchain-core 开始

LangChain 团队已经将框架拆分为多个独立的包，你不再需要一次性引入整个生态。

理解 LangChain 的边界

LangChain 不是万能的。在以下场景中它非常有用：

  • 快速原型开发
  • 标准的 RAG 流程
  • 多工具的 Agent

但在以下场景中它可能不是最佳选择：

  • 极其简单的 LLM 调用（直接用原生 SDK）
  • 有严格延迟要求的实时系统
  • 需要精确控制底层行为的场景

结语

LangChain 的创始人 Harrison Chase 有过一个很好的类比：LangChain 之于 LLM 开发，就像 React 之于前端开发。它提供了一套标准的范式，让复杂的应用可以被有组织地构建出来。

学会 LangChain 不一定让你做出更好的 AI 产品，但它能让你更快地验证想法，更系统地组织代码。在这个 AI 应用形态还在快速演进的阶段，开发效率往往比运行效率更重要。

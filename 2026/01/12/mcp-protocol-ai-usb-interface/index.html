MCP 协议详解：AI 应用的「USB 接口」

Created2026-01-12|Updated2026-05-08
|Word Count:1.6k|Reading Time:5mins|Post Views:

如果把 AI Agent 比作一台电脑，那么 MCP（Model Context Protocol，模型上下文协议）就是它的 USB 接口——一个标准化的、可插拔的外设连接规范。

2024 年 11 月，Anthropic 发布了 MCP 的初始规范。起初很多人没太在意——「又一个 AI 协议而已」。但到 2025 年中，MCP 已经成为 AI 工具集成的事实标准，被 OpenAI、Google、Microsoft 等主流厂商广泛采纳。这背后有着深刻的技术逻辑。

AI 集成之痛：M×N 问题

在没有 MCP 之前，AI 工具集成的生态是碎片化的。

假设有 5 个 LLM 平台（Claude、GPT、Gemini、Llama、Qwen）和 10 个数据源（GitHub、Slack、数据库、文件系统、Google Drive……），要全面覆盖所有组合，需要开发 5×10=50 个独立的集成适配器。这还不算维护成本和升级带来的连锁反应。

这就是典型的 M×N 问题。MCP 的设计目标就是把这个 M×N 问题转化为 M+N 问题：

这和 USB 的历史角色完全一致——在 USB 出现之前，键盘、鼠标、打印机各有各的接口。USB 统一了物理连接层，MCP 统一了 AI 的上下文连接层。

MCP 的架构设计

MCP 采用了经典的 Client-Server 架构，基于 JSON-RPC 2.0 协议：

核心原语（Primitives）

MCP 定义了三种核心交互：

原语 方向 用途 示例
Resources Server → Client 暴露可读取的数据 文件内容、数据库记录、API 响应
Tools Client → Server 暴露可调用的函数 创建 issue、发送邮件、查询天气
Prompts Server → Client 暴露预定义的 prompt 模板 代码审查模板、翻译模板

此外还有 Sampling（Server 请求 Client 调用 LLM，用于 Agent 嵌套场景）和 Transports（定义通信方式）。

传输层：Stdio 与 SSE

MCP 支持两种传输方式：

Stdio（标准输入/输出）

  • Server 作为子进程运行
  • 适合本地工具（文件系统操作、本地数据库）
  • 延迟低，安全性高（天然进程隔离）

SSE（Server-Sent Events）+ HTTP

  • Server 作为独立服务运行
  • 适合远程工具（云端 API、团队共享的工具）
  • 支持多 Client 同时连接

与 Function Calling 的区别

很多人问：MCP 和 OpenAI 的 Function Calling 有什么区别？

Function Calling 是 LLM API 层面的约定——它定义了 LLM 如何「表达」自己想要调用哪个工具。但工具具体如何实现、如何连接，Function Calling 并不关心。

MCP 是工具集成层的协议——它定义了工具如何被发现、描述、调用和管理。MCP 不关心 LLM 内部用 Function Calling 还是其他机制来触发工具调用。

两者是不同层次的东西：Function Calling 工作在 LLM API 层，MCP 工作在工具集成层。一个 MCP Tool 最终可能是通过 Function Calling 来触发的，也可能不是。

MCP 的未来方向

Anthropic 在 MCP 的 Roadmap 中提到了几个重要的演进方向：

  1. Streaming Tools：支持工具调用的流式输出（如大型文件处理进度）
  2. Resource Subscriptions：Client 可以订阅 Resource 的变化通知（如文件被修改）
  3. Remote MCP：标准化的认证和授权机制，让远程 MCP 更加安全
  4. Agent-to-Agent：多个 Agent 之间通过 MCP 互相调用

实践建议

什么时候该写一个 MCP Server？

如果你的 LLM 应用需要访问某个数据源或系统，而这个数据源还没有现成的 MCP Server，你就应该考虑自己写一个。

一个 MCP Server 的开发成本很低（基本就是封装几个函数），但收益很大——写一次，所有 MCP-compatible 的 Client 都可以用。

MCP 的局限

MCP 不是银弹：

  • 它不解决 LLM 本身的推理质量问题
  • 并发和性能还没有深入优化
  • 工具调用的权限粒度还比较粗（整个 Server 要么全用、要么全不用）

结语

MCP 最重要的创新不是技术上的——JSON-RPC 和 Client-Server 都谈不上新——而是生态上的。它解决了一个行业协同问题：让大家在工具集成的层面上「说同一种语言」。

如果一个来自硬件领域的概念可以类比的话：MCP 之于 AI 生态的意义，就像 PCI 总线之于 PC 时代。它不决定你造出什么样的设备，但它让设备之间的连接变得高效和标准化。

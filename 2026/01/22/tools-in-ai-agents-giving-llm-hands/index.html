AI Agent 中的 Tool 系统：让 LLM 拥有双手

Created2026-01-22|Updated2026-05-08
|Word Count:1.8k|Reading Time:6mins|Post Views:

LLM 的推理能力令人惊叹，但它有一个致命的短板——没有「手」。它不能查数据库、不能发邮件、不能操作文件、不能调用 API。Tool（工具）就是给 LLM 装上这些「手」，让它从「会说」变成「能做」。

从 OpenAI 的 Function Calling，到 Anthropic 的 Tool Use，再到 MCP 协议中的 Tool 原语，工具调用正在成为 AI Agent 架构中最核心的环节之一。

工具的本质：一段结构化的描述信息

从技术上讲，一个 Tool 就是一段传递给 LLM 的结构化描述，告诉 LLM：你可以调用这个函数，它的名字、参数和用途如下。

LLM 看到这个描述后，会根据用户的意图决定是否需要调用 send_email，以及用什么参数来调用。注意，LLM 不真正执行这个函数——它只是输出一个结构化的调用请求，你的框架代码负责实际执行。

工具设计的最佳实践

1. 好名字是成功的一半

工具的名字是 LLM 选择调用哪个工具的第一判断依据。名字应该：

  • 清晰描述功能（search_flights 而不是 do_search）
  • 使用一致的命名规范（统一 snake_case 或 camelCase）
  • 避免歧义（get_user vs get_user_profile）

2. Description 要详尽但精炼

好的 description 不只是描述功能，还要暗示使用场景——LLM 通过 description 来理解「什么情况下该用这个工具」。

3. 参数设计：少而精

工具的参数越多，LLM 出错的概率越高。将工具的功能聚焦到单一职责，如果确实需要复杂操作，可以用工作流串联多个工具。

4. 返回结果的结构化

结构化的工具返回有两个好处：

  1. LLM 能更准确地提取信息
  2. 如果你的 Agent 需要串联多个工具调用，结构化的数据更容易被下一个工具消费

工具的层次分类

不是所有的工具都是同质的。在复杂 Agent 中，工具可以按照功能和权限分层：

决策层工具

这些工具不直接操作数据，而是帮助 Agent 决策。比如：

  • route_to_specialist(domain): Agent 将任务路由到特定领域的子 Agent
  • needs_more_info(): Agent 判断当前信息是否足够

操作层工具

这是最大的一类工具——直接操作外部系统或数据的工具。这类工具需要注意两点：

  • 幂等性：创建类的工具（send_email、create_ticket）应该有幂等性设计
  • 错误处理：工具失败时返回明确的错误信息，让 LLM 决定如何重试

基础层工具

计算、日期格式化、单位转换等工具。这些看似简单的工具常常被人忽略——「LLM 自己不会算吗？」实际上，即使是 GPT-4 也可能在复杂计算或日期转换上出错，提供一个确定性的工具来处理这些操作是更安全的做法。

从 Tool 到 Skill

Tool 是最小的可复用单元——一个原子操作。但很多业务场景需要将多个 Tool 组合起来。这就引出了 Skill 的概念。

Skill 是一组相互协作的 Tool + Prompt 的组合，封装了一个完整的能力：

Tool 与 Skill 的关系，就像微服务与 API——原子化的工具可以灵活组合，而 Skill 提供了更高层的抽象。在你构建 Agent 系统时，应该同时拥有这两个层次的思维。

结语

如果我能给 AI 应用开发者一条关于 Tool 的建议，那就是：把 Tool 当作 API 来设计。好的 Tool 设计标准——清晰的命名、明确的职责、结构化的输入输出、考虑安全性——和应用 API 的设计标准是完全一致的。Tool 不是在给 LLM 写函数，而是在给整个 Agent 系统设计接口。
